# 伴随开发审计指南

更新日期：2026-06-02

本文档规定 CyberLab ContentOps 开发过程中如何记录证据、控制风险和更新文档。目标不是增加形式负担，而是让课程实验和后续接手开发的人能够看懂系统为什么这样做、做到了哪一步、还差什么。

## 1. 审计对象

以下变化必须留下文档记录：

- 新增或修改 API。
- 新增趋势数据源或爬虫。
- 新增内容生成器、提示词或模型配置。
- 新增平台适配器或外部命令。
- 新增真实平台动作，例如创建草稿、上传素材、发布内容、评论互动。
- 新增账号、权限、密钥或登录态配置。
- 修改前端核心工作流。
- 修改数据模型或持久化方案。

对应文档入口：

| 变化类型 | 主要文档 |
| --- | --- |
| 架构边界变化 | `architecture.md` |
| 能力升级 | `capability_levels.md` |
| 审计信号变化 | `audit_metrics.md` |
| 平台接入变化 | `platform_status.md` |
| 实验产物变化 | `experiment_artifacts.md` |
| 验证证据变化 | `validation_protocol.md` |
| 命令变化 | `commands.md` |

## 2. 开发前检查

开始开发前先确认：

1. 目标属于课程演示、预览、草稿还是真实发布。
2. 是否涉及外部平台账号或敏感配置。
3. 是否有可用 mock，避免一上来打真实平台。
4. 是否需要新增测试数据或备用数据。
5. 需要同步更新哪些文档。

## 3. 开发中记录

开发过程中至少记录：

- 关键输入：接口请求、平台字段、生成参数。
- 关键输出：响应摘要、预览载荷、草稿路径、日志 ID。
- 关键命令：测试命令、构建命令、外部工具命令摘要。
- 失败路径：错误码、失败原因、是否使用 fallback。
- UI 证据：截图路径、浏览器检查结果、控制台错误情况。

不要记录：

- Cookie。
- 验证码。
- Access Token。
- AppSecret。
- 真实用户隐私数据。
- 未脱敏的平台账号信息。

## 4. 开发后验收

每次交付前检查：

| 项目 | 要求 |
| --- | --- |
| API | 请求体、响应体和错误路径清楚 |
| 后端测试 | 至少覆盖核心服务或接口契约 |
| 前端测试 | 新增可见流程应覆盖主要交互 |
| 浏览器检查 | 新增页面或关键 UI 变化应有截图或手动检查记录 |
| 外部动作 | 默认 preview-only 或 draft-only |
| 日志 | 能看到任务类型、标题、消息、状态和时间 |
| 文档 | 架构、计划或平台状态已同步更新 |

若功能目标是从 L2 升到 L3 或更高，还必须补充 `validation_protocol.md` 中定义的验证包摘要。

## 9. 禁止事项

在没有明确审核机制前，禁止：

- 从前端直接执行外部发布命令。
- 默认自动点击真实平台发布按钮。
- 将真实发布接口暴露为普通按钮。
- 把测试账号登录态提交进仓库。
- 在文档、日志或截图中暴露密钥和 Cookie。
- 用真实账号做高频自动化实验。

## 10. 建议保留的证据

推荐在阶段性交付时保留：

- 测试输出摘要。
- 前端截图路径。
- API 响应摘要。
- 预览载荷样例。
- 外部工具版本或提交号。
- 已知风险和下一步处理计划。
